Lab 3 Reflection – AWS Cloud Security Fundamentals (Capital One Connection)
Summary of Configurations
Identity and Access Management (IAM): I began by verifying that Multi-Factor Authentication was already enabled on the root account, with two MFA devices registered. I then created a restricted IAM group called LabUsers and attached a read-only S3 policy, granting only the minimum permissions necessary for the lab. I created a user called lab-student-user, added them to the LabUsers group, and verified the configuration by successfully signing into the console as that user — confirming limited access was enforced.
Secure Storage (S3): I created a private S3 bucket with Block all public access enabled across all four sub-settings. I confirmed this on the bucket's Permissions tab, which displayed a green checkmark indicating no public access was permitted. I also noted the Bucket Policy section, which provides an additional layer of fine-grained access control.
Secured Compute (EC2): I launched a free-tier t3.micro EC2 instance running Amazon Linux. During configuration I set the Security Group inbound rule to allow SSH only from my specific IP address (99.43.218.115/32), avoiding the dangerously broad 0.0.0.0/0 default. I also checked the instance metadata settings and confirmed IMDSv2 was available, which requires a session token before metadata can be accessed.
Visibility and Governance: I navigated to the Billing and Cost Management dashboard to establish financial awareness and confirm free tier status. I also visited the CloudTrail console, which provides continuous logging of all API activity across the AWS account.

Explicit Capital One Case Study Connection
IAM / Least Privilege: The Capital One breach was significantly worsened by overly permissive IAM credentials attached to a misconfigured WAF role. That role had far broader S3 permissions than it ever needed, allowing the attacker to exfiltrate over 100 million customer records. By creating a restricted LabUsers group with read-only S3 access rather than using the root account or an admin role, I directly demonstrated how least privilege limits the blast radius of stolen credentials. Even if lab-student-user's credentials were compromised, an attacker could do very little damage.
S3 / Block Public Access: In the Capital One incident, sensitive data stored in S3 was accessible because bucket permissions were not properly locked down. Enabling Block all public access on my S3 bucket ensures that no misconfigured bucket policy or ACL can accidentally expose data to the public internet. This single control, had it been uniformly enforced at Capital One, would have prevented the bulk of the data exfiltration.
EC2 / Security Groups and IMDSv2: The attacker exploited a Server-Side Request Forgery vulnerability in Capital One's WAF to query the Instance Metadata Service (IMDSv1) and retrieve temporary IAM credentials. IMDSv1 responded to any request without authentication, making it trivially exploitable via SSRF. By restricting my Security Group to only my IP and enabling IMDSv2 — which requires a session token that an SSRF attack cannot easily obtain — I directly close the attack chain Capital One failed to close. Even if an attacker found an SSRF path into my instance, they could not retrieve metadata credentials.
CloudTrail / Continuous Monitoring: One of the most damaging aspects of the Capital One breach was the detection gap — the attacker operated undetected for months. CloudTrail logs every API call made in an AWS account, meaning actions like unusual S3 GetObject calls at scale would generate log entries. Had Capital One been actively monitoring CloudTrail with alerting, the mass data access would have triggered alarms far earlier. Relying on periodic security audits instead of continuous logging creates exactly the window of opportunity an attacker needs.

Critical Analysis
The S3 and IAM controls felt straightforward to implement correctly because AWS defaults lean toward security — Block public access is on by default, and the console makes least-privilege policy attachment relatively intuitive. Security Groups, however, felt most prone to misconfiguration. The default option during EC2 launch populates SSH as open to 0.0.0.0/0, meaning a user who clicks through without reading would unknowingly expose their instance to the entire internet. This is exactly the kind of misconfiguration that causes real incidents.
Beyond the technical controls, the Capital One breach highlights that tools alone are insufficient. The security controls that would have prevented the breach — least-privilege IAM, IMDSv2, S3 access blocks, CloudTrail monitoring — all existed at the time of the incident. The failure was organizational: no culture of enforcing these defaults, no continuous monitoring practice, and no governance process to catch the WAF misconfiguration before an attacker did. In large environments, security must be embedded into engineering culture and enforced through policy and automation, not left as optional configuration.
